# ADR-0003: 생산 진행 상태를 메뉴 진입 시점 지연 계산으로 갱신

## 상태
Accepted (2026-07-15)

## 배경
생산라인 큐의 항목은 "총생산시간"이 지나면 자동으로 완료 처리되어야 한다(PRD.md FR-5.3). 이 완료 처리를 실제로 언제, 어떻게 트리거할지는 애플리케이션 전체의 실행 모델(백그라운드 스레드 유무, 시간 소스 관리 방식)과 테스트 가능성(NFR-4)에 직접 영향을 준다. 백그라운드 스케줄러를 도입할지, 아니면 시간 경과를 순수하게 조회 시점에 계산할지는 이후 뒤집기 어려운 실행 모델 선택이다.

## 결정
실시간 백그라운드 스케줄러(스레드/타이머)를 두지 않는다. 대신 사용자가 메뉴 4(모니터링)/5(생산라인 조회)/6(출고 처리)에 진입할 때마다 `ProductionService.advance(now_fn())`를 호출해, 큐 맨 앞 항목의 경과시간이 총생산시간을 넘었는지 그 시점에 계산하고 필요한 만큼 완료 처리를 수행한다(PRD.md §5.5 구현 노트, SPEC.md §6.2). 현재 시각은 `now_fn` 함수 인자로 주입한다(DataMonitor `clock_now` DI 패턴 재사용).

## 고려한 대안
1. **백그라운드 스레드/타이머로 실시간 갱신** — `threading.Timer`나 별도 스레드가 주기적으로 큐를 검사해 완료 처리. 실제 서비스라면 자연스러운 방식이나, 콘솔 애플리케이션에서 메인 스레드의 `input()` 블로킹과 동시에 안전하게 상태를 갱신하려면 락(lock) 처리가 필요해지고, pytest에서 결정론적으로 시간을 통제하기 어려워짐(NFR-4 위반 위험).
2. **사용자가 명시적으로 "시간 경과 시뮬레이션" 메뉴를 선택** — 별도 메뉴 항목을 추가해 수동으로 트리거. 테스트하기는 쉬우나 PDF/SPEC에 없는 메뉴를 추가로 만들어야 하고, 사용자가 매번 수동 트리거를 눌러야 하는 번거로움이 있음.
3. **메뉴 진입 시점 지연 계산(채택)** — PRD.md §5.5가 이미 "실시간 백그라운드 스케줄러 없이 사용자가 메뉴에 진입할 때마다 갱신"이라고 명시.

## 선택 이유
- PRD.md/SPEC.md가 이미 이 방식을 구현 노트로 명시하고 있어, 별도 스케줄러를 도입하는 것은 명세를 벗어나는 과설계다.
- 시간 소스(`now_fn`)를 함수 인자로 주입할 수 있게 하면, 테스트에서 고정된 `datetime`을 넘겨 생산 진행률/완료 시점을 결정론적으로 검증할 수 있다(NFR-4).
- 백그라운드 스레드가 없으므로 메인 스레드의 `input()` 대기 중 상태 변경 경합을 걱정할 필요가 없다 — 단일 프로세스 콘솔 앱이라는 전제(PRD.md §3 Out of Scope: 동시성 제어 없음)와 일치한다.

## 장점
- 스레드 동기화, 락, 백그라운드 예외 처리 등 콘솔 앱에는 과한 복잡도를 도입하지 않는다.
- 시간 소스 주입 덕분에 순수 함수에 가까운 형태로 pytest 단위 테스트 작성이 쉬움.
- "메뉴 진입 시 최신 상태로 반영"이라는 사용자 경험이 PDF 예시 화면과 자연스럽게 일치한다.

## 단점
- 사용자가 오랫동안 관련 메뉴(4/5/6)에 진입하지 않으면, 실제로는 생산이 완료되었어도 화면상 반영이 지연된다(다만 메뉴 재진입 시 큐 앞부터 순차적으로 모두 갱신되므로 최종 데이터 정합성은 깨지지 않는다).
- 메인 메뉴 요약 정보(등록 시료 종수, 생산라인 대기 건수 등, SPEC.md §1)를 정확히 보여주려면 메인 메뉴 진입 시에도 동일한 `advance()` 호출이 필요 — 갱신 트리거 지점을 빠뜨리면 화면 간 수치 불일치가 생길 수 있다.

## 후속 영향
- 신규 메뉴/화면이 생산 상태(재고, 주문 상태)를 노출한다면, 그 진입 지점에서도 `ProductionService.advance(now_fn())`를 호출하도록 컨트롤러 구현 체크리스트에 포함해야 한다.
- 향후 실시간성이 실제로 요구되면(예: 웹 대시보드로 확장) 이 ADR을 재검토하고 백그라운드 갱신 방식을 별도로 설계해야 한다.
