# ADR-0004: 승인 시점 재고 예약을 파생값으로 계산

## 상태
Accepted (2026-07-15)

## 배경
동일 시료를 참조하는 주문 A(수량 100, 시료 재고 60)가 승인되면 부족분 40에 대해 생산 큐가 등록되고 `PRODUCING`으로 전환되지만, `sample.stock` 자체는 승인 시점에 차감하지 않는다(SPEC.md §4.2, PRD.md FR-3.2 — 차감은 출고 시점, FR-6.1). 이 상태에서 같은 시료의 주문 B(수량 60)를 승인 판정하면, 기존 `preview_approval`이 raw `sample.stock=60`만 보고 "재고 충분"으로 즉시 `CONFIRMED` 처리한다. 그러나 그 60은 이미 A가 확보하기로 되어 있던 수량이다. 이후 B가 먼저 출고되어 `stock=0`이 되고, A의 생산이 완료되어 `stock=40`이 된 뒤 A가 출고되면 `stock=40-100=-60`으로 재고가 음수가 된다.

근본 원인은 승인 판정이 "이미 다른 주문이 확보한 수량"을 전혀 고려하지 않는다는 것이다. 즉 재고 예약 개념이 없다.

## 결정
승인 판정(FR-3.2)에서 raw `sample.stock` 대신 **가용 재고**(`stock - 예약 재고`)를 사용한다. 예약 재고는 동일 시료를 참조하는 다른 주문 중 이미 `CONFIRMED`/`PRODUCING`으로 승인 확정된 주문의 `quantity` 합이며, **저장되는 새 필드가 아니라 매 판정 시점에 계산하는 파생값**이다(`calculations.reserved_qty()`). 아직 승인 여부가 결정되지 않은 `RESERVED` 상태 주문은 예약 재고에 포함하지 않는다 — 승인 전 주문까지 재고를 선점하면 나중 승인 심사가 불필요하게 보수적으로 막히기 때문이다.

`sample.stock` 필드 자체의 차감/증가 시점(승인 시 변화 없음 · 생산 완료 시 `+=` · 출고 시 `-=`)은 그대로 유지한다 — SPEC.md §4.2/PRD.md FR-3.2에 명시된 "재고는 승인 시점에 차감하지 않는다"는 불변식을 바꾸지 않는다.

추가로 `ShippingService.release()`에도 차감 직전 `stock >= quantity` 방어 검증을 추가한다. 예약 재고 기반 승인 판정이 근본 원인을 막지만, 두 검증은 서로 다른 시점(승인 vs 출고)의 위험을 막는 별개의 방어선이므로 어느 한쪽만으로 줄이지 않는다.

## 고려한 대안
1. **`Sample`에 `reserved` 필드를 추가해 승인/거절/생산완료/출고마다 갱신** — 예약량이 항상 최신 상태로 저장되어 매번 다시 계산할 필요가 없다. 그러나 `samples.json` 스키마 변경과 기존 데이터 마이그레이션이 필요하고, 갱신 누락 시 저장된 값과 실제 주문 상태가 어긋나는 이중 관리 위험이 생긴다.
2. **승인 시점에 재고를 즉시 차감(예약분만큼)** — 재고 예약을 "차감"으로 흡수하면 별도 예약 로직이 필요 없다. 그러나 SPEC.md §4.2/PRD.md FR-3.2에 이미 명시된 "재고는 승인 시점에 차감하지 않는다(차감은 출고 시)"는 불변식과 정면으로 충돌한다.
3. **파생값으로 매 판정 시점에 계산(채택)** — 이미 존재하는 `pending_demand(orders, sample_id)` 패턴(`MonitoringService.stock_report()`가 소비, FR-4.2)과 동일한 방식으로, 대상 상태만 다르게(`RESERVED+PRODUCING+CONFIRMED` → `CONFIRMED+PRODUCING`) 재사용한다.

## 선택 이유
- 스키마 변경/마이그레이션이 전혀 필요 없다 — 기존 `Sample`/`samples.json` 구조를 그대로 둔다.
- "재고는 승인 시점에 차감하지 않는다"는 이미 확정된 불변식(SPEC.md §4.2)을 지킨다.
- `calculations.py`에 이미 존재하는 "상태 집합을 대상으로 quantity 합을 계산" 패턴(`pending_demand`)을 그대로 확장하는 것이라 새로운 개념을 도입하지 않는다.
- FIFO 원칙과 자연스럽게 일치한다 — 먼저 승인된 주문이 그 시점의 가용 재고를 먼저 차지하고, 나중 승인 판정은 이미 선점된 수량을 제외한 나머지만 본다.

## 장점
- 데이터 마이그레이션 없이 버그를 수정할 수 있다.
- `reserved_qty`/`pending_demand`가 물리적으로 다른 함수로 분리되어 있어, FR-3.2(승인 판정, 커밋된 예약)와 FR-4.2(모니터링, 전망 지표)의 목적 차이가 코드에서도 드러난다.
- `ShippingService.release()`의 방어 검증과 함께, 승인 판정 로직에 남아있을 수 있는 미지의 경로로 인한 재고 음수화까지 이중으로 막는다.

## 단점
- 매 승인 판정마다 `order_repo.list_all()`로 전체 주문을 조회해 합산하므로, 주문 수가 매우 많아지면 `pending_demand()`와 마찬가지로 O(n) 비용이 든다(현재 단일 프로세스 콘솔 앱 규모에서는 문제 되지 않음).
- 동시성 제어가 없는 전제(PRD.md §3 Out of Scope) 위에서 성립한다 — 여러 프로세스가 동시에 승인을 처리하는 환경이라면 이 파생값 계산만으로는 경쟁 상태를 막을 수 없다.

## 후속 영향
- 향후 재고 예약을 실제 필드로 승격해야 할 정도로 계산 비용이 문제가 되거나, 다중 프로세스 동시성 제어가 필요해지면 이 ADR을 재검토해야 한다.
- 새로운 주문 상태나 재고 관련 판정 로직을 추가할 때는 `reserved_qty`(승인 판정용, 커밋된 수량)와 `pending_demand`(모니터링용, 전망 수량) 중 어느 지표를 써야 하는지 목적에 맞게 선택해야 한다 — 두 지표를 하나로 통합하지 않는다.
