# ADR-0001: JSON 파일 기반 영속성 채택

## 상태
Accepted (2026-07-15)

## 배경
시스템은 Sample/Order 데이터를 애플리케이션 재시작 후에도 유지해야 한다(PRD.md NFR-1). 저장소 형식을 무엇으로 할지는 이후 Repository/Service 계층 전체의 인터페이스, 테스트 방식, 배포 방식을 좌우하는 선택이며, 한 번 데이터가 쌓이기 시작하면 다른 형식으로 옮기는 비용이 커진다.

## 결정
`data/samples.json`, `data/orders.json`, `data/production_queue.json` 세 개의 JSON 파일에 각각 레코드 리스트 전체를 저장하고, 매 쓰기마다 임시 파일에 쓴 뒤 `os.replace()`로 원자적 치환한다(`JsonFileStore`, DataPersistence PoC 재사용). 조회/수정 시 파일 전체를 메모리로 읽고 다시 통째로 쓰는 방식을 취하며, 별도의 트랜잭션이나 부분 갱신, 동시 접근 제어는 두지 않는다.

## 고려한 대안
1. **SQLite** — 파일 기반이면서도 트랜잭션과 인덱스, 동시성 제어를 제공.
2. **인메모리(프로세스 종료 시 소실)** — 구현이 가장 단순하지만 NFR-1(영속성)을 충족하지 못함.
3. **외부 DB 서버(PostgreSQL 등)** — 다중 사용자/동시 접속을 고려할 때 유리하나 배포·운영 복잡도가 급증.
4. **JSON 파일 + 원자적 쓰기(채택)** — 미션1 4개 PoC가 이미 이 방식으로 검증됨.

## 선택 이유
- 미션1의 4개 PoC(ConsoleMVC, DataPersistence, DataMonitor, DummyDataGenerator)가 모두 동일한 JSON 스키마와 `JsonFileStore` 패턴으로 이미 검증되어 있어, 재구현/재검증 비용 없이 그대로 재사용 가능하다.
- PRD.md §3에서 다중 사용자 동시 접속, 실제 DB 서버 연동을 명시적으로 Out of Scope로 규정하고 있어, SQLite/외부 DB가 주는 이점(동시성, 트랜잭션)이 이 프로젝트 범위에서는 실제로 쓰이지 않는다.
- 콘솔 단일 프로세스 앱이라는 규모에서 JSON 파일은 별도 드라이버·마이그레이션 도구 없이 사람이 직접 열어 확인할 수 있어 디버깅/시연에 유리하다.

## 장점
- 별도 의존성(DB 드라이버, 스키마 마이그레이션 도구) 없이 표준 라이브러리(`json`, `os`)만으로 구현 가능.
- 파일을 직접 열어 텍스트로 확인·수정할 수 있어 개발/디버깅/데모 시연이 쉬움.
- PoC에서 이미 검증된 원자적 쓰기(`os.replace`)로 쓰기 도중 프로세스 중단에도 데이터 손상을 방지.
- pytest에서 `tmp_path`로 격리된 파일을 사용해 테스트하기 쉬움(NFR-8).

## 단점
- 매 쓰기마다 파일 전체를 다시 씀 — 레코드 수가 커지면(수만 건 이상) 성능이 선형으로 저하됨.
- 동시 프로세스 접근 시 데이터 경합/유실 위험 — 다중 사용자 지원으로 확장하려면 재설계가 필요.
- 쿼리 기능(조건 검색, 정렬, 집계)을 전부 애플리케이션 코드(Repository/Service)에서 직접 구현해야 함 — DB의 인덱스/쿼리 최적화 이점을 포기.

## 후속 영향
- 향후 "다중 사용자 동시 접속" 또는 "대량 데이터" 요구가 생기면 이 ADR을 재검토하고 SQLite/DB로의 마이그레이션을 별도 ADR로 다뤄야 한다.
- Repository 인터페이스(`create/get/list_all/list_by_status/update` — `delete`는 현재 스펙상 필요 없어 미구현)를 CRUD 형태로 유지해 두면, 저장소를 교체하더라도 Service 계층 이상은 영향받지 않도록 설계했다(ARCHITECTURE.md §1 레이어 구조 참고).
